# OmniClaw Semantic Memory 🧠 (ChromaDB Segment)

**Path:** `brain/chroma_db/`
**Namespace:** `brain.chroma_db`
**Technology:** ChromaDB v1.5.6 + HNSW Algorithmic Search

This directory holds one of the most critical structural paradigms of **OmniClaw V5.0 Blueprint**: the persistent, multi-dimensional semantic vector database.

## 🚀 What Makes This Folder Outstanding?

Unlike standard SQL or NoSQL databases that search by exact keyword matches, this folder gives OmniClaw the ability to "think" and "associate" concepts by **meaning**.

### 1. The HNSW Graph Engine (Hierarchical Navigable Small World)
Look closely at the UUID directory (e.g., `48aea028.../`). Inside, it houses binary files like `data_level0.bin`. This isn't raw text—it is a hyper-dimensional spatial map. Each ingested knowledge node is evaluated by the `all-MiniLM-L6-v2` neural network and assigned exactly **384 mathematical coordinates** (a 384-dimension float32 matrix).
When the system needs to recall information, it doesn't scan words; it calculates the *spatial distance* between ideas in 384-D space using HNSW.

### 2. The Hybrid Persistence Model (`chroma.sqlite3`)
The whopping ~193MB `chroma.sqlite3` file acts as the backbone bridging semantic search and deterministic retrieval. While the `.bin` files handle raw spacial awareness, the SQLite DB handles:
*   Structured filtering (e.g., "Find this concept, but only if generated by `CIV_FETCHED` today").
*   FTS (Full-Text Search) fallback indices mapping back to metadata.
*   Transactions and queues ensuring thread-safe ingestion by `OA_Academy`.

### 3. Automated Intake Pipelines
This folder is the endpoint for multiple dynamic agent pipelines:
*   **CIV Intake:** Live link ingestion mapped directly from the CEO terminal (`aos intake`).
*   **Orphan Sweep:** Automated background daemon jobs finding and embedding lost knowledge files.

### 4. Git LFS Optimization
Despite hosting hundreds of megabytes of raw database files, this folder is elegantly structured within the repository via a highly specific **Git LFS (Large File Storage)** rule, allowing seamless GitHub synchronization without choking bandwidth or cluttering conventional commit history.
