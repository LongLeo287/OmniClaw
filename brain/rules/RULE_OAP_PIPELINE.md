# OmniClaw Assimilation Pipeline (OAP)

The following pipeline is the highest law of data and repository assimilation within OmniClaw. It guarantees 100% full-stack autonomous execution without human intervention. Core Daemons MUST run asynchronously via their respective command panels or background Watchdogs.

## The 6-Daemon Golden Pipeline

### 1. Gate 1: OIW (Intake Harvester)
- **Role:** The Plow & Harvester.
- **Rules:** OIW reads from queue lists (e.g. `repos_pending.txt`). It uses advanced context compression (like `gitingest`) to crush full repositories into minimal text blocks.
- **Lineage Tracing:** If a repository is pre-vetted by a human (CIV Report), OIW MUST prefix the output directory with `CIV_FETCHED_`. Otherwise, `FETCHED_`.
- **Destination:** Raw data is shoved into `vault/tmp/sandbox`.
- **Tools Used:** Local caching, CLI parsing (`gitingest` / recursive concatenators).

### 2. Gate 2: OSF (Firewall & Sentinel)
- **Role:** The Security Inspector.
- **Rules:** OSF blindly monitors `sandbox/`. It does NOT care about code topology. It runs regex heuristics targeting hardcoded secrets, API Keys, and traces of malignant logic. It runs dependency audits (Dependabot logic).
- **Destination:** Passes clean structures to `vault/tmp/quarantine`. Malicious directories are annihilated or sandboxed into `quarantine/FAILED`.
- **Tools Used:** Yara rules, Regex scanners.

### 3. Gate 3: OHD (Doctor & Linter)
- **Role:** Deep Code Sanity & Structuring.
- **Rules:** Monitors `quarantine/`. Automatically lints and normalizes raw Python/JS files if required. Ensures minimum base structure before LLMs read it.
- **Destination:** Pushes healed repositories to `vault/tmp/raw_knowledge_dumps`.
- **Tools Used:** Black/Flake8 (Python), Prettier, Regex healing.

### 4. Gate 4: OA (Academy & Supreme Elder)
- **Role:** Contextual Understanding & Component Forging.
- **Rules:** Monitors `raw_knowledge_dumps/`. 
- **Deep-Sea Directive:** If OA spots a `CIV_FETCHED_` prefix, it invokes extreme timeouts (7200s) and large buffers (80,000 chars) to comprehensively map the architecture.
- **Action:** Triggers the LLM to write physical code, blueprints, and explicitly categorizes the drop as a Skill, Agent, Workflow, or API. It must draft an `UPGRADE_PROPOSAL.md`.
- **Destination:** Outputs physical crafted folders to `vault/tmp/state_queues/OER_INBOX`.
- **Fallbacks:** If API fails, triggers native fallback `_DIR_IDENTITY.md` creation.

### 5. Gate 5: OER (Registry & Placement)
- **Role:** Traffic Controller & Database Writer.
- **Rules:** Monitors `OER_INBOX`. Reads identity metadata or infer type via folder properties.
- **Federated Registration (MAPPING/GRAPHING):** OER is strictly responsible for updating the distributed map network. It strips prefixes (`FETCHED_` / `CIV_FETCHED_`), generates unique IDs, and automatically inserts the entity into its corresponding shard in `brain/indices/FAST_{TYPE}_INDEX.json`.
- **Destination:** Physically transports the folder to its everlasting domain (`ecosystem/workforce/agents`, `ecosystem/skills`, `ecosystem/plugins`, etc).
- **Graph Embedding:** OER hooks into Vector DB to push knowledge for autonomous AI relations (Graphing).

### 6. Gate 6: OMA (Architect & Memory Tracker)
- **Role:** System Nervous System.
- **Rules:** Sweeps the `ecosystem/` natively. Enforces the strict rule that NO directory exists without an Identity (`_DIR_IDENTITY.md`).
- **Action:** Merges the shards generated by OER to construct the master `OMA_SYSTEM_MAP.json`, guaranteeing a unified architecture where Agents can dynamically locate one another without hardcoding paths.

## Execution Rules (Auto-Mapping & Zero-Trust Compliance)
- **Zero-Trust Quarantine (Luật Không Cấp Phép Bừa Bãi):** ONLY vetted, complete, and source-verified packages are allowed to be mapped. If a raw file or folder lacks a clear origin lineage (`CIV_FETCHED` vs `FETCHED`), has malicious fragments, or fails completion checks, it MUST BE QUARANTINED (`vault/tmp/quarantine/FAILED_ASSIMILATE`). OER and OMA are explicitly FORBIDDEN from registering, assigning IDs, or mapping unknown entities.
- **No File Drop Exceptions:** Clean raw data dumped directly into `ecosystem/workforce` will be automatically targeted by OHD/OMA/OA for in-place compliance enforcement.
- **Complete Profile Generation:** No entity is allowed to exist without an `AGENT.md`/`SKILL.md` (Capability), a `_DIR_IDENTITY.md` (Identity), and an entry in `FAST_{TYPE}_INDEX` (Map). If missing but the source is trusted, OA will be auto-triggered to generate them using Local LLM.
- No Daemon may skip a gate. Censorship (Gate 2 - OSF) takes precedence over all other logic.
- No human intervention is required once `OIW` catches a URL, provided it passes OSF.
- Local AI Models (Ollama/Waiters) act as primary failovers for API collapse. OA forces execution regardless of collapse.
