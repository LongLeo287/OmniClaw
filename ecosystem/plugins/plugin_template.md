# OmniClaw Standard Plugin Template

> **Purpose:** This template provides the standard structural foundation for any new plugin assimilated into the OmniClaw System 2 Ecosystem.

## 1. Directory Layout

When establishing a new plugin, the directory structure must strictly follow:

```text
ecosystem/plugins/[plugin_name_snake_case]/
├── __init__.py                # Package marker (Required for Python imports)
├── _DIR_IDENTITY.md           # Mandatory system routing and taxonomy tag
├── PLUGIN.md                  # Auto-generated capability profile (by OA Academy)
├── requirements.txt           # Explicit dependencies for this plugin
├── main.py / app.py           # Entry point logic
└── tests/                     # Local test suites
```

## 2. Standard `_DIR_IDENTITY.md` Format

This file is automatically generated by `OAP Intake Pipeline`, but if establishing a plugin manually, use this structure:

```yaml
---
id: [plugin_name]
type: plugin
owner: [author/maintainer]
created_at: [timestamp]
---

# [Plugin Name]

Brief description of the plugin's execution capability.
```

## 3. Deployment Hook

Always ensure your core functions expose a standardized initialization hook so `OMA` can bridge it to the Workforce agents.

```python
# __init__.py
def initialize_plugin(context):
    """
    Hook called by Core Daemons during boot sequence.
    Returns standard interface dictionary.
    """
    pass
```

## 4. Zero-Trust Rules
- A plugin **must not** attempt to write files into `core/daemons` or `core/ops`.
- A plugin **must** store its persistent states or large data dumps strictly in the `vault/tmp` or `vault/assets` layer.
- Ensure all API Keys drop to `.env` file at the root, do NOT hardcode secrets in plugin code.
